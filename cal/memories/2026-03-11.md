## Meeting: GitHub Project Board Integration

**Date:** 2026-03-11
**Participants:** Cal + user

### Key Insights
- Issues are for tracking state, not storing artifacts. Specs, minutes, implementation notes stay in the repo.
- Three-level hierarchy: Release > Epic > Feature. Two boards, one milestone mechanism.

### Decisions Made
- **Three levels of work tracking** — Rationale: mirrors real product shipping (release = App Store submission, epic = feature suite, feature = one shippable thing). Confidence: high.
- **Two GitHub Project boards per repo:**
  - **Epics board** — Columns: Idea → In Progress → Shipped
  - **Features board** — Columns: Cal → Lisa → Ralph → QA → Cleanup
  - Rationale: clean separation of granularity. Confidence: high.
- **Releases = GitHub Milestones** — group Epics, no separate board. Rationale: releases are just lists of epics, don't need kanban. Confidence: high.
- **Cal creates and moves everything** — issues created at end of `/cal:meet`, moved at every `/cal:next` approval gate. User can override on GitHub. Confidence: high.
- **Minimal issue bodies** — title, one-liner, link to parent. No artifact accumulation. Rationale: artifacts live in repo (`docs/specs/`, `cal/cal.md`). Confidence: high.
- **Epic auto-advancement** — moves to In Progress when first Feature enters Features board, moves to Shipped when all Features clear Cleanup. Confidence: high.
